(D2) **push activé**, nouvel event `"reponse"`, destinataire = l'autre membre ; (D3) badges minimaux sur réponse (seul « ✎ Édité »). **Schéma** : colonne `parent_id` auto-référente nullable (`on delete cascade`) — migration `20260627120000_sprint23_fridge_note_replies.sql` ; trigger `enforce_fridge_note_single_level` (`search_path=''`) garde l'invariant **au niveau données** : parent doit être une note de tête, même foyer, et une note **ayant des réponses ne peut pas devenir une réponse** (bloque le re-parentage qui créerait Q→P→C, durcissement issu de la revue) ; **R7** : le payload ne porte que le type d'événement, jamais le corps. **Correctif de revue** : la pastille « non lues » de l'accueil (`app/page.tsx`) filtrait sans `parent_id is null` — une réponse (jamais marquable, D1) figeait le compteur → corrigé. **Preuve d'acceptation observable** (Chromium réel sur `/demo/frigo`, 0 erreur JS) : 4 notes de tête montrent « Répondre » ; ### Sprint 24 — Épingler une note du frigo (une seule par foyer, auteur seul) ✅
**Feature dirigée par l'auteur, suite Sprints 20-23** (prolonge le tableau du frigo : une note de tête peut rester **épinglée en haut**, cas d'usage liste d'épicerie permanente). **Décisions produit (auteur, en session)** : portée **auteur seul** (on n'épingle/détache que sa propre note) ; **une seule épingle par foyer** (épingler en détache la précédente) ; **pas de push** (action de tri, faible signal). **Réconciliation d'abord — prémisse de la carte CORRIGÉE** : la carte proposait « simple colonne + petite UPDATE, **pas de RPC** ». Vérifié en session : c'était **faux** — la policy `fridge_notes_update` durcie au Sprint 21 (`20260626130000_sprint21_fridge_update_no_receipt_forge.sql:18-23`) impose `read_at/read_by` **nuls** dans son `with check` → un `UPDATE … set is_pinned` direct sur une note **déjà lue** échouerait (ou forcerait un reset d'accusé erroné) ; et « une épingle par foyer » exige de détacher l'épingle de l'**autre** membre, interdit par la RLS auteur-seul. **Livré** : migration `20260627130000_sprint24_fridge_note_pin.sql` — colonne `is_pinned` (défaut false), CHECK `fridge_notes_pin_head_only` (`not (is_pinned and parent_id is not null)` → une réponse n'est jamais épinglée), **index unique partiel** `idx_fridge_notes_one_pin (household_id) where is_pinned` (rempart BD tous chemins), et RPC **`epingler_note_frigo(note_id, pin)` SECURITY DEFINER** (calque `marquer_note_frigo_lue`) : vérifie **membre du foyer** (parité — un membre **révoqué** garde author_id, sans cette garde il pourrait encore épingler par appel RPC direct), **auteur seul**, **tête seulement**, puis détache l'épingle existante du foyer avant de poser la nouvelle. Le RPC ne touche **ni `body` ni `read_at`** → le trigger Sprint 22 ne bumpe pas `updated_at` (pas de faux « Édité ») et l'accusé n'est jamais réinitialisé. **Threading** : `is_pinned`/`isPinned` ajouté via la source unique `FRIGO_NOTE_COLUMNS` + schéma Zod + `parseFrigoRows` (`lib/frigo/db-rows.ts`), `FrigoNote.isPinned` (`lib/frigo/types.ts`), `lib/database.types.ts` (hand-edit fidèle + signature RPC). **Tri à deux niveaux** dans `grouperEnFils` : épingle d'abord (`Number(b.isPinned) - Number(a.isPinned)`), puis récent-d'abord. **Action** `epinglerNoteFrigo` (`app/frigo/actions.ts`, RPC ; pas de push) ; **UI** (`components/frigo/tableau-frigo.tsx`) : bouton 📌 Épingler/Détacher auteur-seul sur les notes de tête + badge « 📌 Épinglée » visible des **deux** membres, mise à jour optimiste (une épingle/foyer reflétée localement) ; i18n `fr.frigo.{epingler,detacher,epinglee,erreurEpinglage}`. **R7 intact** : `is_pinned` est un drapeau non sensible. **Preuve d'acceptation observable** (Chromium réel sur `/demo/frigo`, **0 erreur JS**) : la note de l'**autre** membre pré-épinglée (seed) montre le badge mais **aucun** bouton bascule (auteur seul) ; épingler **ma** note → elle **remonte en tête** avec le badge ET l'épingle précédente est **détachée** (une seule épingle, cross-auteur) ; **aucune réponse** n'a de bouton d'épingle (tête seulement). **Revue indépendante** `/code-review` high (8 angles, contexte frais) : 1 finding **confirmé** corrigé — le RPC omettait la garde `is_household_member` du RPC frère (un membre **révoqué** aurait pu épingler ses anciennes notes par appel direct) → garde ajoutée + test ; reste = refus vérifiés (Realtime livre toujours la ligne courante complète → pas de réversion d'épingle ; bouton aligné sur ses pairs non-`disabled`) ; passe `/simplify` : 0 changement (code aligné Sprints 20-23). Gates mesurés : vitest **300** (+12 : 7 isolation RLS frigo **contre un vrai Postgres** via `scripts/local-db.sh` — auteur épingle / non-auteur refusé / une épingle cross-auteur / réponse refusée / index unique / non-bump updated_at + accusé préservé / **membre révoqué refusé** ; +2 `grouperEnFils` épingle-d'abord ; +3 `parseFrigoRows` is_pinned), tsc 0, biome 0, build OK. **Tâche opérationnelle** (hors session) : appliquer la migration Sprint 24 sur Cloud + régénérer les types. ### Sprint 25 — Listes d'épicerie dans le frigo (partagées, cochage à deux, push anti-spam) ✅
**Feature dirigée par l'auteur** (définie en session ; hors FR-1→17, consignée). Prolonge l'espace du frigo (Sprints 20-24) vers un nouvel usage : des **listes d'épicerie** où **les deux membres** créent une liste, ajoutent/retirent des articles et **cochent/décochent** ce qui est acheté. **Réconciliation d'abord** (loi anti-hallucination) : tout le code épicerie était **à créer** (aucune table/type/action `epicerie` n'existait — vérifié) ; le helper RLS `is_household_member` (`supabase/migrations/20260611192623_rls_policies.sql:14`) et l'infra push (`push_subscriptions`, VAPID, Resend, gabarit `notify-fridge`) **existaient** et ont été réutilisés (aucun nouveau secret/env). **Décisions produit (auteur, en session)** : (1) **gestion à deux** (ajouter/retirer/cocher ouverts aux deux membres) ; (2) **placement** = section dédiée « Épicerie » dans `/frigo`, séparée du tableau de notes ; (3) **push au cochage anti-spam** (le 1ᵉʳ cochage d'une liste notifie le conjoint, puis silence pendant une fenêtre de cooldown). **Schéma** : 2 tables (migration `20260628120000_sprint25_grocery_lists.sql`) — `grocery_lists` (titre, `last_check_notified_at` = débounce **interne**, jamais sélectionné/forgeable côté client) et `grocery_items` (`is_checked`/`checked_by`/`checked_at`, `household_id` dénormalisé pour RLS+Realtime, CHECK de cohérence). **RLS partagée** : SELECT/INSERT/DELETE = membre du foyer ; INSERT durci (coche non forgeable) ; trigger `enforce_grocery_item_household` (anti-injection cross-foyer, INSERT+UPDATE). **Realtime** : les 2 tables publiées (garde d'existence comme `fridge_notes`). **Couche pure** : `lib/epicerie/{types,db-rows}.ts` — `GROCERY_LIST_COLUMNS`/`GROCERY_ITEM_COLUMNS` (source unique), frontière Zod `parseGrocery*Rows`, helpers `grouperListes`/`compteRestant`. **Actions** `app/epicerie/actions.ts` (`getUser()` + RLS + `revalidatePath`) ; push best-effort via `notify-grocery-client.ts` → Edge `notify-grocery` (calque `notify-fridge` ; destinataire = l'autre membre ≠ acteur). **R7** : payload `grocery-payload.ts` + copie Deno (parité testée) ne portent **que** le type d'événement, jamais un libellé d'article. **UI** `components/epicerie/listes-epicerie.tsx` (cases ☑/☐ ≥ 44 px icône+texte, libellé barré + « coché par X », badge « X à acheter », mise à jour optimiste, Realtime), intégrée à `app/frigo/page.tsx` ; i18n `fr.epicerie`. **Preuve d'acceptation observable** (Chromium réel sur `/demo/epicerie`, **0 erreur JS applicative** — seul un 404 favicon bénin, confirmé par curl) : liste seedée « 2 à acheter », article « Pain » coché « par Conjoint(e) » ; cocher « Lait » → « 1 à acheter », décocher → « 2 à acheter » ; création d'une 2ᵉ liste + ajout d'article. **Revue indépendante** `/code-review` high (8 angles, contexte frais) : **0 bug de correction / 0 faille RLS / 0 violation CLAUDE.md** confirmés (candidats réfutés : `upsertItem` n'affecte pas l'ordre car `grouperListes` re-trie ; `revalidatePath("/")` inutile, pas d'indicateur épicerie sur l'accueil) ; passe `/simplify` : 1 import mort `useRef` retiré (duplications `functionsBaseUrl`/garde d'auth gardées — parité voulue avec le frigo).
