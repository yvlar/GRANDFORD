### Sprint 21 — Édition d'une note du frigo par son auteur ✅
**Feature dirigée par l'auteur, suite Sprint 20** (complète le CRUD du tableau du frigo : créer/lire/retirer → **modifier**). **Réconciliation d'abord** (loi anti-hallucination) : la policy RLS `fridge_notes_update` (auteur seul) **existe déjà** depuis Sprint 20 (`supabase/migrations/20260626120000_sprint20_fridge_notes.sql:59-61`, préparée « pour l'édition future du corps ») → **aucune migration de schéma nécessaire**. **Décisions produit (auteur, en session)** : (1) l'édition **réinitialise l'accusé de lecture** (le contenu a changé → la note redevient « non lue », `read_at/read_by` → null) ; (2) push **uniquement si la note était déjà lue** (contenu périmé à re-signaler), event `"modifiee"`. **Livré** — server action `modifierNoteFrigo` (`app/frigo/actions.ts`, calquée sur `supprimerNoteFrigo` : `getUser()`, Zod `frigoBodySchema`+`uuidSchema`, double-garde `.eq("author_id")`, lit l'état lu AVANT pour décider du push, UPDATE qui **hardcode `read_at/read_by` à null** — reset voulu ET non-forge d'accusé) ; UI d'édition inline auteur-seul dans `components/frigo/tableau-frigo.tsx` (bouton ✏️ à côté de Retirer, textarea préremplie auto-focus, Enregistrer/Annuler, cibles ≥ 44 px, icône+texte NFR-12) ; nouvel event push `"modifiee"` propagé au miroir TS↔Deno (`lib/notifications/fridge-payload.ts` + `supabase/functions/_shared/fridge-payload.ts`, parité gardée) et à l'Edge `notify-fridge` (validation + destinataire = l'autre membre, comme `"nouvelle"`). **Correctif d'interaction** (trouvé en session) : le handler Realtime UPDATE retire l'id du garde anti-tempête `dejaMarquees` quand `readAt` repasse à null, sinon une note rééditée réafficherait « Nouveau » chez le conjoint sans jamais redevenir « lue » (boucle édition → re-lecture → accusé restaurée). **Durcissement RLS issu de la revue indépendante** : la policy UPDATE Sprint 20 laissait l'auteur **forger** un accusé non-nul sur sa propre ligne (asymétrie avec la policy INSERT déjà durcie) → migration `20260626130000_sprint21_fridge_update_no_receipt_forge.sql` resserre le `with check` (`read_at is null and read_by is null`) ; invariant net : un accusé n'est posé QUE par le RPC `marquer_note_frigo_lue` (SECURITY DEFINER, exempt de RLS), effacé QUE par une édition de l'auteur. **Preuve d'acceptation observable** (Chromium réel sur `/demo/frigo`, 0 erreur JS) : clic ✏️ sur une note à soi déjà lue (« Lu ») → textarea préremplie du corps actuel → Enregistrer → **nouveau corps affiché** + note repassée à **« Pas encore lu »** (reset constaté) ; la note de l'autre membre **n'a aucun bouton ✏️** (auteur seul). **Revue indépendante** `/code-review` high (contexte frais) : 1 MAJOR (forge d'accusé par l'auteur) corrigé par la migration + 1 test ; 3 mineurs appliqués (auto-focus textarea, reset du bandeau d'erreur à l'ouverture, démo conserve `createdAt` pour ne pas réordonner) ; passe `/simplify` : 0 changement (code aligné sur les patterns Sprint 20). Gates mesurés : vitest **265** (+8 : 6 isolation RLS frigo **contre un vrai Postgres** via `scripts/local-db.sh` — auteur édite / non-auteur bloqué / reset d'accusé / non-forge auteur / non-reset non-auteur / CHECK corps vide ; +2 payload `"modifiee"`), tsc 0, biome 0, build OK. **Tâche opérationnelle** (hors session) : appliquer la migration Sprint 21 sur Cloud (cf. tâches en attente) ; redéployer `notify-fridge` (event `"modifiee"`). Branche : `claude/prompt-executer-sprint-2f5v2g`.

### Sprint 22 — Indicateur « Édité » de note du frigo ✅
**Feature dirigée par l'auteur, suite Sprint 21** (distingue une note **rééditée** d'une note neuve). **Réconciliation d'abord (loi anti-hallucination) — prémisse de la carte CORRIGÉE** : la carte proposait « badge si `updatedAt > createdAt` », **sans schéma**, en s'appuyant sur la colonne `updated_at` existante (Sprint 20). Vérifié en session : c'était **faux** — le trigger partagé `set_updated_at` (`supabase/migrations/20260611192620_initial_schema.sql:18`) bumpe `updated_at` à **chaque** UPDATE, et le RPC d'accusé `marquer_note_frigo_lue` (`20260626120000_sprint20_fridge_notes.sql:103-109`) fait un UPDATE de `read_at/read_by` → une note simplement **lue** aurait `updated_at > created_at` et serait faussement « Éditée ». STOP + signalement ; l'auteur a choisi la version **corrigée**. **Livré** : un trigger **propre à `fridge_notes`** ne bumpe `updated_at` **que si `body` change** — `set_fridge_notes_updated` (`set search_path = ''`), migration `20260626140000_sprint22_fridge_edited_trigger.sql` ; le trigger partagé `set_updated_at` reste **intact sur les 9 autres tables** (vérifié). `now()` étant stable en transaction, une note fraîche a `updated_at == created_at` exactement → une lecture ne bouge plus rien, seule une vraie édition avance `updated_at`. **Aucun nouveau champ** (`updated_at` existait déjà, présent dans `lib/database.types.ts`). **Threading** : `updatedAt` ajouté à `FrigoNote` (`lib/frigo/types.ts`), à `parseFrigoRows` + son schéma Zod (`lib/frigo/db-rows.ts`) et aux **3 SELECT** de notes complètes (`app/frigo/page.tsx`, `app/frigo/actions.ts` ×2) ; le handler Realtime réutilise `parseFrigoRows`, la colonne arrive donc automatiquement. **Helper pur** `estEditee(note)` (`lib/frigo/types.ts`) : comparaison en **temps epoch** (`getTime()`, robuste aux offsets de fuseau), jamais lexicographique. **UI** : badge discret « ✎ Édité » (icône+texte, NFR-12) dans le pied de carte, visible des **deux** membres ; i18n `fr.frigo.edite`. **R7 intact** : `updated_at` est un horodatage non sensible ; aucun motif/paye/corps de note ne fuit. **Preuve d'acceptation observable** (Chromium réel sur `/demo/frigo`, 0 erreur JS hors 404 d'asset bénin) : la note éditée (`updatedAt > createdAt`) affiche **✎ Édité** ; la note **lue mais non éditée** (« ✓ Lu ») n'affiche **PAS** « Édité » (preuve : lire ≠ éditer) ; les notes fraîches sans badge. **Revue indépendante** `/code-review` high (3 angles finder, contexte frais) : **0 finding** (shared trigger non modifié, `search_path` sûr, INSERT donne `created==updated`, payload Realtime complet, types déjà présents, conventions OK) ; passe `/simplify` : code déjà propre. Gates mesurés : vitest **272** (+7 : 3 `estEditee` + 2 `parseFrigoRows` + 2 isolation RLS — lecture ne bumpe pas `updated_at` / édition l'avance — **contre un vrai Postgres** via `scripts/local-db.sh`), tsc 0, biome 0, build OK. **Migration appliquée sur Cloud** (post-fusion, via Supabase MCP) : trigger `set_fridge_notes_updated` actif en prod, lecture ne pose plus « Édité » (vérifié) ; au passage, la RLS owner-only de `cycle_templates` (**Sprint 14, manquante sur Cloud**) a été appliquée. Branche : `claude/prompt-executer-sprint-ngjtyp` (PR #50 → `dev`, release #51 → `main`).

### Sprint 23 — Réponses au frigo (fil à un seul niveau) ✅
**Feature dirigée par l'auteur, suite Sprints 20-22** (complète le tableau du frigo vers un canal léger : un membre peut **répondre** à une note — **un seul niveau**, une réponse ne se répond pas). **Réconciliation d'abord** (loi anti-hallucination) : prémisse **propre** — `fridge_notes` n'avait **aucune** colonne `parent_id` (`supabase/migrations/20260626120000_sprint20_fridge_notes.sql:14-25`), `FrigoNote`/`parseFrigoRows` vérifiés (`lib/frigo/types.ts:10-18`, `lib/frigo/db-rows.ts:8-32`). **Décisions produit (auteur, en session)** : (D1) **pas d'accusé de lecture sur les réponses** (une réponse EST l'accusé) ; (D2) **push activé**, nouvel event `"reponse"`, destinataire = l'autre membre ; (D3) badges minimaux sur réponse (seul « ✎ Édité »). **Schéma** : colonne `parent_id` auto-référente nullable (`on delete cascade`) — migration `20260627120000_sprint23_fridge_note_replies.sql` ; trigger `enforce_fridge_note_single_level` (`search_path=''`) garde l'invariant **au niveau données** : parent doit être une note de tête, même foyer, et une note **ayant des réponses ne peut pas devenir une réponse** (bloque le re-parentage qui créerait Q→P→C, durcissement issu de la revue) ; index partiel `idx_fridge_notes_parent` ; RPC `marquer_note_frigo_lue` resserré (`and f.parent_id is null` → refuse un accusé sur une réponse, D1). **RLS inchangée** : les 4 policies portent déjà sur `household_id`+`author_id`, valables pour une ligne de réponse ; le `with check` durci (Sprint 21, `read_*` nuls) garantit déjà une réponse sans accusé. **Helper pur** `grouperEnFils` (`lib/frigo/types.ts`) regroupe la liste plate en fils (têtes récentes d'abord, réponses chronologiques, orphelines écartées) ; la règle D1 vit dans `statutLecture`/`estNouvellePourMoi` (réponse → « non-applicable »/false) pour que **tout** consommateur (badge d'accueil, auto-marquage) la respecte sans double garde. **Threading** : `parentId` ajouté à `FrigoNote`, `parseFrigoRows`+schéma Zod, et aux SELECT via la constante partagée `FRIGO_NOTE_COLUMNS` (un seul endroit à toucher). **Actions** : `repondreNoteFrigo` (`app/frigo/actions.ts`) dérive `household_id` de la note parente lue sous RLS (zéro confiance client), insère `parent_id`, push `"reponse"`. **UI** (`components/frigo/tableau-frigo.tsx`) : fils mémoïsés, réponses indentées sous leur note de tête, composeur de réponse inline (focus au montage), « Répondre » sur les notes de tête seulement (ouvert aux deux membres) ; `CarteNote` dérive « est une réponse » de `note.parentId`. **Push** : event `"reponse"` propagé au miroir TS↔Deno (`lib/notifications/fridge-payload.ts` + `supabase/functions/_shared/fridge-payload.ts`, parité gardée) + validation Edge `notify-fridge` ; **R7** : le payload ne porte que le type d'événement, jamais le corps. **Correctif de revue** : la pastille « non lues » de l'accueil (`app/page.tsx`) filtrait sans `parent_id is null` — une réponse (jamais marquable, D1) figeait le compteur → corrigé. **Preuve d'acceptation observable** (Chromium réel sur `/demo/frigo`, 0 erreur JS) : 4 notes de tête montrent « Répondre » ; la réponse semée s'affiche **indentée** dans `ul[aria-label="réponses"]` et **n'a aucun** « Répondre » (un seul niveau) ; le composeur poste une **nouvelle réponse indentée** sous son parent. **Revue indépendante** `/code-review` high (8 angles, contexte frais) : 1 bug confirmé corrigé (pastille d'accueil) + altitude (règle D1 poussée dans les helpers) + durcissement trigger (re-parentage) + exclusivité des modes édition/réponse ; passe `/simplify` (4 angles) : `FRIGO_NOTE_COLUMNS` partagé, `estReponse` dérivé de `note.parentId` (prop retirée), `grouperEnFils` mémoïsé. Gates mesurés : vitest **288** (+16 : 4 `grouperEnFils` + 5 `parseFrigoRows`/parentId + 7 isolation RLS frigo — réponse même foyer / cross-foyer refusée / un seul niveau (insert + re-parentage UPDATE) / mismatch foyer / cascade / RPC refuse une réponse — **contre un vrai Postgres** via `scripts/local-db.sh`), tsc 0, biome 0, build OK. **Tâche opérationnelle** (hors session) : appliquer la migration Sprint 23 sur Cloud + régénérer les types ; redéployer `notify-fridge` (event `"reponse"`). Branche : `claude/prompt-executer-sprint-gyj97n`.

### Sprint 24 — Épingler une note du frigo (une seule par foyer, auteur seul) ✅
**Feature dirigée par l'auteur, suite Sprints 20-23** (prolonge le tableau du frigo : une note de tête peut rester **épinglée en haut**, cas d'usage liste d'épicerie permanente). **Décisions produit (auteur, en session)** : portée **auteur seul** (on n'épingle/détache que sa propre note) ; **une seule épingle par foyer** (épingler en détache la précédente) ; **pas de push** (action de tri, faible signal). **Réconciliation d'abord — prémisse de la carte CORRIGÉE** : la carte proposait « simple colonne + petite UPDATE, **pas de RPC** ». Vérifié en session : c'était **faux** — la policy `fridge_notes_update` durcie au Sprint 21 (`20260626130000_sprint21_fridge_update_no_receipt_forge.sql:18-23`) impose `read_at/read_by` **nuls** dans son `with check` → un `UPDATE … set is_pinned` direct sur une note **déjà lue** échouerait (ou forcerait un reset d'accusé erroné) ; et « une épingle par foyer » exige de détacher l'épingle de l'**autre** membre, interdit par la RLS auteur-seul. **Livré** : migration `20260627130000_sprint24_fridge_note_pin.sql` — colonne `is_pinned` (défaut false), CHECK `fridge_notes_pin_head_only` (`not (is_pinned and parent_id is not null)` → une réponse n'est jamais épinglée), **index unique partiel** `idx_fridge_notes_one_pin (household_id) where is_pinned` (rempart BD tous chemins), et RPC **`epingler_note_frigo(note_id, pin)` SECURITY DEFINER** (calque `marquer_note_frigo_lue`) : vérifie **membre du foyer** (parité — un membre **révoqué** garde author_id, sans cette garde il pourrait encore épingler par appel RPC direct), **auteur seul**, **tête seulement**, puis détache l'épingle existante du foyer avant de poser la nouvelle. Le RPC ne touche **ni `body` ni `read_at`** → le trigger Sprint 22 ne bumpe pas `updated_at` (pas de faux « Édité ») et l'accusé n'est jamais réinitialisé. **Threading** : `is_pinned`/`isPinned` ajouté via la source unique `FRIGO_NOTE_COLUMNS` + schéma Zod + `parseFrigoRows` (`lib/frigo/db-rows.ts`), `FrigoNote.isPinned` (`lib/frigo/types.ts`), `lib/database.types.ts` (hand-edit fidèle + signature RPC). **Tri à deux niveaux** dans `grouperEnFils` : épingle d'abord (`Number(b.isPinned) - Number(a.isPinned)`), puis récent-d'abord. **Action** `epinglerNoteFrigo` (`app/frigo/actions.ts`, RPC ; pas de push) ; **UI** (`components/frigo/tableau-frigo.tsx`) : bouton 📌 Épingler/Détacher auteur-seul sur les notes de tête + badge « 📌 Épinglée » visible des **deux** membres, mise à jour optimiste (une épingle/foyer reflétée localement) ; i18n `fr.frigo.{epingler,detacher,epinglee,erreurEpinglage}`. **R7 intact** : `is_pinned` est un drapeau non sensible. **Preuve d'acceptation observable** (Chromium réel sur `/demo/frigo`, **0 erreur JS**) : la note de l'**autre** membre pré-épinglée (seed) montre le badge mais **aucun** bouton bascule (auteur seul) ; épingler **ma** note → elle **remonte en tête** avec le badge ET l'épingle précédente est **détachée** (une seule épingle, cross-auteur) ; **aucune réponse** n'a de bouton d'épingle (tête seulement). **Revue indépendante** `/code-review` high (8 angles, contexte frais) : 1 finding **confirmé** corrigé — le RPC omettait la garde `is_household_member` du RPC frère (un membre **révoqué** aurait pu épingler ses anciennes notes par appel direct) → garde ajoutée + test ; reste = refus vérifiés (Realtime livre toujours la ligne courante complète → pas de réversion d'épingle ; bouton aligné sur ses pairs non-`disabled`) ; passe `/simplify` : 0 changement (code aligné Sprints 20-23). Gates mesurés : vitest **300** (+12 : 7 isolation RLS frigo **contre un vrai Postgres** via `scripts/local-db.sh` — auteur épingle / non-auteur refusé / une épingle cross-auteur / réponse refusée / index unique / non-bump updated_at + accusé préservé / **membre révoqué refusé** ; +2 `grouperEnFils` épingle-d'abord ; +3 `parseFrigoRows` is_pinned), tsc 0, biome 0, build OK. **Tâche opérationnelle** (hors session) : appliquer la migration Sprint 24 sur Cloud + régénérer les types.
