### Sprint 18 — Garde de parité du miroir de format de date ✅
**Dette technique tirée des SPRINTS SUGGÉRÉS** (carte réconciliée avec le code réel : toutes références vérifiées en session, prémisse valide). La date civile `'AAAA-MM-JJ'` est formatée par **deux** `Intl.DateTimeFormat` strictement identiques mais dupliqués — `FORMAT_JOUR_LONG` (`lib/schedule/format.ts:14`, affichage app) et `FORMAT_JOUR` (`lib/notifications/payload.ts:26`, texte des rappels FR-10) — la duplication étant imposée par la contrainte **zéro-import** de `payload.ts` (consommé par l'Edge Function Deno, hors alias `@/`). Chaque fichier portait le commentaire réciproque « MIROIR ASSUMÉ », mais **rien ne gardait** que les deux produisent la même chaîne : modifier un formateur sans l'autre faisait diverger en silence la date affichée de celle écrite dans la notification (même classe de risque que la dérive du corps du payload corrigée au Sprint 16). **Livré** : `lib/notifications/format-parity.test.ts` (nouveau, pur, aucun mock) — `FORMAT_JOUR` étant privé, sa surface observable est `reminderPayload(date, "day").body` ; le test asserte l'**égalité exacte** `body === \`Écart à l'horaire ${FORMAT_JOUR_LONG.format(dateUTC(date))} (demain).\`` sur 5 dates couvrant les axes du format (bornes d'année 01-01/12-31, avant l'ancre Pitman, 29 février bissextile, date ordinaire). **Revue indépendante** `/code-review` high (2 angles, contexte frais) : 1 finding confirmé — l'assertion initiale en `toContain` était **uni-directionnelle** (une dérive qui *élargit* `FORMAT_JOUR`, p. ex. ajout de `year`, laissait la chaîne plus courte de l'app incluse → faux négatif) ; corrigé en égalité exacte → garde **bidirectionnelle**. **Preuve que le test mord** (constatée en session) : drift `month:"short"` → rouge ; drift *additif* `year:"numeric"` → rouge (capté par l'égalité exacte, raté par l'ancien `toContain`) ; restauré → vert. Aucune migration BD, aucun changement moteur, aucun changement de payload ni de code de production. Gates mesurés : vitest **224** (+5 ; 147 passés + 77 isolation RLS ignorés faute de Postgres local), tsc 0, biome 0. Branche : `claude/prompt-executer-sprint-526ypa`.

### Sprint 19 — Interrupteur de la Fenêtre de sommeil ✅
**Feature dirigée par l'auteur** (réponse en session ; cadre FR-6, sommeil). **Réconciliation d'abord** (loi anti-hallucination) : le pick naturel des SPRINTS SUGGÉRÉS — « garde de parité d'une copie Edge des échéances » — reposait sur une prémisse **fausse** → STOP. Vérifié en session : aucun miroir TS↔TS non gardé n'existe (`payload.ts` et le format de date sont déjà gardés, Sprints 16/18) ; `lib/notifications/echeances.ts` n'a **pas** de copie Edge — `send-reminders/index.ts:66` lit des `remind_at` **pré-calculés**, la duplication des offsets `30/7/1` est TS↔SQL, déjà gardée par `supabase/tests/reminders.test.ts`. Direction re-choisie par l'auteur : **activer/désactiver la Fenêtre de sommeil**. **Livré** : le travailleur peut éteindre l'affichage de sa fenêtre de sommeil. **Modèle** : colonne `sleep_defaults.enabled` (boolean, défaut `true`) — migration `20260625120000_sprint19_sleep_enabled.sql` ; **disponibilité partagée** (policy `sleep_defaults_all` inchangée — la conjointe la lit, **aucun motif en jeu, R7 intact**) ; heures conservées même désactivé (réactiver = retrouver sa fenêtre). **Moteur pur** : paramètre **en queue** `sleepEnabled = true` sur `dayStatuses`/`statusForDate`/`monthGrid` (`lib/schedule/status.ts`) ; la garde tient en une ligne — `sleeping = sleepEnabled && shift === null && previousShift === "nuit"` ; désactivé, la récupération redevient `conge` (travailleur) / `disponible` (conjointe), `sleep: null`. Défaut `true` ⇒ **zéro régression** (tous les call-sites et tests existants inchangés). **Frontière** : `parseSleepEnabled` (`lib/schedule/db-rows.ts`, parseur frère de `parseSleepRow` — défaut `true` sans ligne). **UI** : case à cocher accessible (libellé + texte, jamais la couleur seule — NFR-12 ; cible ≥ 44 px) dans `components/sommeil/fenetre-sommeil.tsx` ; server action `definirFenetreSommeil` persiste l'état (`active` absent = case décochée = désactivé, sémantique HTML). **Câblage** : `app/page.tsx` lit `enabled` dans le **chemin partagé** (worker ET conjointe), passe `sleepEnabled` à `VueCoupDoeil` (threadé aux 4 appels moteur) ; `/foyer` préremplit la case. **Types** : `enabled` ajouté à `lib/database.types.ts` (hand-edit fidèle au générateur, ordre alphabétique — régénération Cloud par l'auteur, pg_graphql absent du Postgres local). **Preuve d'acceptation observable** (constatée en session, dev SSR HTTP 200) : `/demo/horaire?equipe=C&date=2026-06-05` → pastille **« Sommeil » + plage 07:00–15:00** ; même URL `&sommeilActif=0` → pastille **« Congé », aucune plage** (param démo ajouté). **Revue indépendante** (2 angles contexte frais — correction + R7/conventions/cross-file) : **0 finding** ; passe `/simplify` (2 angles) : 1 cleanup appliqué (ternaire imbriqué aplati dans `fenetre-sommeil.tsx`), reste bien nivelé. Gates mesurés : vitest **232** (+8 : 6 moteur interrupteur + 2 isolation RLS, **exécutés contre un vrai Postgres** via `scripts/local-db.sh` — 35 tests RLS), tsc 0, biome 0, build OK. Branche : `claude/prompt-executer-sprint-klvtk1`.

### Sprint 20 — Note du frigo (tableau partagé, accusés de lecture, push) ✅
**Feature dirigée par l'auteur, hors FR-1→17** (consignée : pas dans `docs/analyse/`). **Réconciliation d'abord** (loi anti-hallucination) : la feature est **réellement neuve** — les « notes » FR-8 (`public.notes`, `components/notes/note-du-jour.tsx`) sont **par date civile**, sans accusé de lecture ; aucun push event-driven ni Realtime n'existait (push **cron-only** via `send-reminders`). **Livré** : nouvelle page `/frigo` — tableau de notes libres où **les deux membres** postent, **accusé de lecture** notifié à l'auteur quand l'autre lit, **seul l'auteur retire**. **Schéma** : table `fridge_notes` (migration `20260626120000_sprint20_fridge_notes.sql`) — accusé plat `read_at`/`read_by`, policies **SPLIT** (SELECT/INSERT/UPDATE membre, **DELETE auteur seul**), INSERT durci `read_* null` (forger un accusé impossible). **Accusé inter-membre** par RPC `marquer_note_frigo_lue` **SECURITY DEFINER** (membre + non-auteur + idempotent ; pas d'UPDATE libre du corps). **Live** : table publiée dans `supabase_realtime` (garde d'existence → migration sûre sur le Postgres local sans la publication) ; composant client `tableau-frigo.tsx` s'abonne aux `postgres_changes` (INSERT/UPDATE/DELETE filtrés foyer), auto-marquage à l'affichage, « Lu ✓ » live. **Push device event-driven** : Edge `notify-fridge` (calque `send-reminders`) déclenchée best-effort par la server action (`fetch`+service-role, timeout 2,5 s, échec jamais bloquant) ; payload pur `fridge-payload.ts` **R7-structurel** (ne reçoit QUE le type d'événement, jamais le corps) + miroir Deno gardé par `fridge-payload-parity.test.ts` (lignée Sprints 16/18). **UI/UX (NFR-12)** : cartes « collées au frigo », icône+texte (jamais couleur seule), cibles ≥44 px, badge « Nouveau », lien d'accueil à pastille. **Revue indépendante** `/code-review` high (3 angles, contexte frais) : RLS/R7 **étanche** confirmé ; corrigés — `.then()`→`async/await`+`useTransition`, parser Realtime réutilise `parseFrigoRows` (frontière Zod unique), accusé optimiste par fusion ciblée (anti-écrasement d'un UPDATE concurrent), anti-tempête de RPC, filtre foyer sur DELETE, INSERT durci, timeout push, couplage env documenté ; passe `/simplify` : `creerNoteFrigo` réutilise `parseFrigoRows`, helper mort retiré. **Preuve d'acceptation observable** (dev SSR, HTTP 200) : `/demo/frigo` (monte le **vrai** `TableauFrigo`, `realtimeActif=false`) montre pose, retrait auteur-seul, badge « Nouveau », et les états « Lu ✓ » / « Pas encore lu ». Gates mesurés : vitest **257** (+25 : 13 isolation RLS **contre un vrai Postgres** via `scripts/local-db.sh` + 6 types + 6 payload/parité), tsc 0, biome 0, build OK. **Tâches opérationnelles** (hors session) : déployer `notify-fridge` + secrets + `SUPABASE_SERVICE_ROLE`/URL functions Vercel ; appliquer la migration sur Cloud + régénérer les types. ### Sprint 21 — Édition d'une note du frigo par son auteur ✅
**Feature dirigée par l'auteur, suite Sprint 20** (complète le CRUD du tableau du frigo : créer/lire/retirer → **modifier**). **Réconciliation d'abord** (loi anti-hallucination) : la policy RLS `fridge_notes_update` (auteur seul) **existe déjà** depuis Sprint 20 (`supabase/migrations/20260626120000_sprint20_fridge_notes.sql:59-61`, préparée « pour l'édition future du corps ») → **aucune migration de schéma nécessaire**. **Décisions produit (auteur, en session)** : (1) l'édition **réinitialise l'accusé de lecture** (le contenu a changé → la note redevient « non lue », `read_at/read_by` → null) ; (2) push **uniquement si la note était déjà lue** (contenu périmé à re-signaler), event `"modifiee"`. **Livré** — server action `modifierNoteFrigo` (`app/frigo/actions.ts`, calquée sur `supprimerNoteFrigo` : `getUser()`, Zod `frigoBodySchema`+`uuidSchema`, double-garde `.eq("author_id")`, lit l'état lu AVANT pour décider du push, UPDATE qui **hardcode `read_at/read_by` à null** — reset voulu ET non-forge d'accusé) ; UI d'édition inline auteur-seul dans `components/frigo/tableau-frigo.tsx` (bouton ✏️ à côté de Retirer, textarea préremplie auto-focus, Enregistrer/Annuler, cibles ≥ 44 px, icône+texte NFR-12) ; nouvel event push `"modifiee"` propagé au miroir TS↔Deno (`lib/notifications/fridge-payload.ts` + `supabase/functions/_shared/fridge-payload.ts`, parité gardée) et à l'Edge `notify-fridge` (validation + destinataire = l'autre membre, comme `"nouvelle"`). **Correctif d'interaction** (trouvé en session) : le handler Realtime UPDATE retire l'id du garde anti-tempête `dejaMarquees` quand `readAt` repasse à null, sinon une note rééditée réafficherait « Nouveau » chez le conjoint sans jamais redevenir « lue » (boucle édition → re-lecture → accusé restaurée). **Durcissement RLS issu de la revue indépendante** : la policy UPDATE Sprint 20 laissait l'auteur **forger** un accusé non-nul sur sa propre ligne (asymétrie avec la policy INSERT déjà durcie) → migration `20260626130000_sprint21_fridge_update_no_receipt_forge.sql` resserre le `with check` (`read_at is null and read_by is null`) ; invariant net : un accusé n'est posé QUE par le RPC `marquer_note_frigo_lue` (SECURITY DEFINER, exempt de RLS), effacé QUE par une édition de l'auteur. **Preuve d'acceptation observable** (Chromium réel sur `/demo/frigo`, 0 erreur JS) : clic ✏️ sur une note à soi déjà lue (« Lu ») → textarea préremplie du corps actuel → Enregistrer → **nouveau corps affiché** + note repassée à **« Pas encore lu »** (reset constaté) ; la note de l'autre membre **n'a aucun bouton ✏️** (auteur seul). **Revue indépendante** `/code-review` high (contexte frais) : 1 MAJOR (forge d'accusé par l'auteur) corrigé par la migration + 1 test ; 3 mineurs appliqués (auto-focus textarea, reset du bandeau d'erreur à l'ouverture, démo conserve `createdAt` pour ne pas réordonner) ; passe `/simplify` : 0 changement (code aligné sur les patterns Sprint 20). Gates mesurés : vitest **265** (+8 : 6 isolation RLS frigo **contre un vrai Postgres** via `scripts/local-db.sh` — auteur édite / non-auteur bloqué / reset d'accusé / non-forge auteur / non-reset non-auteur / CHECK corps vide ; +2 payload `"modifiee"`), tsc 0, biome 0, build OK. **Tâche opérationnelle** (hors session) : appliquer la migration Sprint 21 sur Cloud (cf. tâches en attente) ; redéployer `notify-fridge` (event `"modifiee"`). Branche : `claude/prompt-executer-sprint-2f5v2g`.
